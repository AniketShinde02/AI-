# 07 – Architecture Documentation

This document serves as the high-level source of truth for the system architecture, logical components, deployment patterns, and operational standards.

---

## 1. System Overview (Context)

The system is a voice-first AI assistant composed of the following primary domains:

1.  **Frontend (Next.js)**: The user interface, handling real-time voice streaming and chat UI via GetStream.
2.  **Backend (Next.js API Routes)**: The "brain" hosted in the cloud, orchestrating tokens, DB persistence (Neon Postgres + Drizzle), and high-level tool execution.
3.  **Local Windows Agent (Python)** *(Future / v2)*: A lightweight service running on the user's machine to execute PC-level commands (app control, file system) securely.

### Communication Flow
- **User ↔ Frontend**: Voice/Chat via GetStream SDKs.
- **Frontend ↔ Backend**: REST API for token generation and state management.
- **Backend ↔ AI Services**: Integration with Groq / OpenAI (via custom base URLs for speed).
- **Backend ↔ Stream Edge**: Server-side orchestration of Stream calls and agents.

---

## 2. Voice & Chat Pipeline (GetStream)

The assistant uses **GetStream** for all real-time communication. This avoids building WebRTC signaling from scratch and leverages Stream's edge network.

### 2.1 Voice Flow (MVP Now)
- **Transport**: Stream Video & Audio SDK provides the WebRTC bridge for low-latency voice calls.
- **OpenAI Realtime Integration**: The backend creates a Stream `audio_room` and connects an OpenAI Realtime agent to it. The frontend joins the call using credentials generated by the backend.
- **Backend Token Generation**: The Next.js backend generates user tokens (`generateUserToken`) using the Stream Node SDK to authenticate the frontend securely.
- **Voice Session Flow**: 
  1. Frontend requests session `/api/stream/voice-session`.
  2. Backend ensures the user exists (`upsertUsers`), creates a call, and optionally attaches an AI agent.
  3. Frontend joins the call via Stream SDK.

### 2.2 Text Chat Flow (MVP Now)
- **Transport**: Stream Chat SDK for React.
- **Text Message Sync**: Chat history is persisted to our own Neon Postgres DB (via Drizzle) in parallel with Stream Chat (or primarily via our DB for complete control), while Stream handles real-time delivery.
- **Agent Responses**: The backend receives user messages, calls the LLM, and persists/streams the response back to the client.

### 2.3 Free-Tier Constraints (MVP Now)
- **Video/Audio**: ~333,000 participant minutes free + $100 monthly credit. Focus on audio-only to conserve bandwidth and cost.
- **Chat**: 1,000 MAU and 100 concurrent WebSocket connections on the Build tier.
- **Rule of Thumb**: Re-use persistent "Workspace" calls for solo-dev usage rather than creating thousands of one-off calls.

## 3. Logical Architecture

### 3.1 Backend Structure (Monorepo)
The backend follows a clean layered architecture using Next.js App Router:
- **API Layer**: `src/app/api/...` handles incoming requests, auth, and token generation.
- **Service Layer**: Decoupled integrations for GetStream, LLMs, DB (Postgres via Drizzle).

### 3.2 Background Tasks & State
- **Async Tasks (MVP Now)**: Simple asynchronous API routes (e.g., streaming LLM responses).
- **Job Queue / Async Tasks (Future / v2)**: Celery + Redis or Inngest for background summarization and heavy tool execution without blocking the voice loop.
- **Agent Orchestration (Future / v2)**: LangGraph for multi-step agent flows and stateful reflection.
- **Vector DB (Optional alternative)**: pgvector for long-term semantic memory.

## 4. Production-Grade Guidance

To ensure the backend is stable under load and doesn't spam APIs:

### 4.1 Rate Limiting & Request Budgeting
- **Rate Limiting (MVP Now)**: Throttle token refreshing and API calls on the frontend (e.g., using `sessionStorage` and debouncing).
- **Rate Limiting (Future / v2)**: Redis-based token bucket per user/IP on the API routes to prevent abuse.
- **Request Budgeting**: Assign a `max_token_budget` and `max_steps` per LLM interaction to prevent infinite agent loops.

### 4.2 Caching
- **Caching (MVP Now)**: Aggressive use of React Query (TanStack) for deduping frontend requests. Cache Stream tokens in `sessionStorage` to avoid regenerating them on every mount.
- **Caching (Future / v2)**: Redis layer for expensive database queries or identical LLM lookups.

### 4.3 Retries and Backoff
- **Resilience (MVP Now)**: Simple try-catch logic in API routes.
- **Retries and Backoff (Future / v2)**: Implement exponential backoff for transient external API failures (OpenAI, Groq, Stream).
- **Idempotency**: All tool-calling tasks and DB inserts should handle retries safely without duplicating data.

### 4.4 Observability and Tracing
- **Logging (MVP Now)**: Structured JSON console logs with correlation IDs.
- **Tracing (Future / v2)**: LangSmith or Langfuse to trace complex agent thoughts, tool calls, and measure latency.
- **Metrics (Future / v2)**: Datadog or Prometheus for tracking P95 latency and success rates.

### 4.5 Feature Flags / Remote Config
- **Staged Rollouts (Future / v2)**: Guard non-trivial features (like new agent personas) with remote config flags. Allows shipping "dark" without immediate exposure.
